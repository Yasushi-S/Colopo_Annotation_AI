院内ネットワーク限定

## 開発環境（全プロジェクト共通）

- アプリ種別　: Flask（Python）製 Web アプリ
- アクセス方法: 職場のローカルネットワーク内の複数端末から Google Chrome でアクセス
- 実行環境　　: Windows PC、Python 仮想環境（venv）
- 自動起動　　: Windows タスクスケジューラにより電源投入時に自動起動
- リポジトリ　: GitHub で管理

## 開発フロー（必ず守ること）

```
フェーズ1: 現状把握
  ├─ 新規開発 → ヒアリング
  └─ 既存改修 → コード調査

フェーズ2: 指示書作成
  ├─ 新規開発 → 作成発注書（CURSOR_INSTRUCTIONS.md）
  └─ 既存改修 → 修正指示書（modification_instructions.md）

フェーズ3: ユーザー承認（必須）

フェーズ4: Cursor コーディング → modification_log.md に記録

フェーズ5: Claude Code レビュー

フェーズ6: ユーザー動作確認 → git commit・push
```

---

## フェーズ1: 現状把握

### 既存改修の場合 — コード調査

- `app.py`（メインファイル）を読み、ルート一覧・全体構造を把握する
- 改修対象の関連ルート・テンプレート・モデルを特定する
- 既存の `modification_instructions.md` と `modification_log.md` を読み、過去の経緯を把握する
- **コードを読まずに指示書を作成してはならない**
- **推測でコードの動作を説明してはならない**
- 調査後、影響範囲をユーザーに共有する

---

## フェーズ2: 指示書作成

### 既存改修 — 修正指示書（modification_instructions.md）

末尾に追記する。新規ファイルを作成しないこと。

```
---
## [YYYY-MM-DD] 修正指示 #N ── <一行要約>

### 改修の背景・目的
（なぜ必要か）

### 影響範囲
- 対象ファイル: （ファイルパスを列挙）
- 影響する既存機能: （明記する）
- DB スキーマ変更: あり／なし（ありの場合は詳細を記載）

### 修正内容詳細
（箇条書きで具体的・明確に）

### 注意事項
（デグレードリスク・Cursor が確認すべき事項）
---
```

---

## フェーズ4: Cursor の役割

1. 発注書・修正指示書の内容に従い忠実にコーディングを行う
2. DB スキーマ・既存データへの変更を伴う場合は、必ず作業前にユーザーへ確認を取る
3. コーディング完了後、`modification_log.md` に追記する（末尾追記、新規ファイル作成禁止）

```
---
## [YYYY-MM-DD] 実行 #N ── <一行要約>

### 対応した指示
- （発注書または modification_instructions.md の #N）

### 実施内容詳細
（実際に行った変更を箇条書きで記載）

### 確認事項・備考
（問題があれば記載、なければ「なし」）
---
```

---

## フェーズ5: レビュー観点

- 指示書の内容がすべて実装されているか
- 既存機能が壊れていないか（デグレード確認）
- セキュリティ問題がないか（SQL インジェクション・XSS 等）
- 既存コードのスタイルと一貫しているか

## 必須ファイル（全プロジェクト共通）

### .gitignore

最低限以下を含めること：

```
venv/
__pycache__/
*.pyc
.env
*.db
instance/
.DS_Store
static/uploads/
static/cases/
```

### 起動用 .bat ファイル（start_app.bat）

```bat
@echo off
cd /d "%~dp0"
call "%~dp0venv\Scripts\activate.bat"
python app.py
```

※ デバッグ時は末尾に `pause` を追加可。本番運用時は不要。

### .cursor/rules（Cursor ルールファイル）

プロジェクトルートの `.cursor/rules` を作成すること（CURSOR_INSTRUCTIONS.md に内容を明記）。

### タスクスケジューラ登録手順

- トリガー　: コンピューターの起動時
- 操作　　　: start_app.bat を実行
- 全般設定　: 最上位の特権で実行にチェック、ユーザーがログオンしていなくても実行

---

## 禁止事項

- コードを読まずに指示書を作成すること
- ユーザー承認なしに git commit・push すること
- `.env` ファイルの読み書き・コミット
- 影響範囲が不明なまま指示書を作成すること
- DB スキーマ変更時、作業前のバックアップ・ユーザー確認を省略すること

---

## セキュリティ要件

### 共通（全アプリ）

- `SECRET_KEY` は `.env` で管理し、コードにハードコードしない
- `app.run(debug=True)` は開発時のみ。本番では `debug=False`
- SQL は SQLite + パラメータバインドを使用し、文字列結合による SQL 構築を禁止
- テンプレートでは `{{ }}` による自動エスケープを維持し、`| safe` フィルタを安易に使用しない
- ファイルアップロード機能は拡張子・MIMEタイプを検証し、アップロード先を `static/` 直下に置かない
- 依存パッケージは `requirements.txt` で固定し、既知の脆弱性があるバージョンを使用しない

### 院内ネットワーク限定アプリ

- アクセス元 IP をローカルネットワーク（例: `192.168.x.x`）に制限することを検討する
- 操作ログ（誰がいつ何をしたか、特に承認操作）を記録する
- `start_app.bat` でバインドアドレスを `0.0.0.0` にする場合は、ファイアウォールで外部アクセスを遮断する
